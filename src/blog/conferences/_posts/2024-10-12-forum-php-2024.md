### Create your own malware sandbox - Alexandre Daubois

Explication PENTESTS. 
Un rapport de pentest lui a indiqué des soucis dans l'upload de fichiers par les clients du site.
3 options :
- Abandonner

Comment ça marche ?

#### Static heuristic method

- Pattern detection 
- Metadata inspection

#### Dynamtic heuristic Method

- Sandbox execution
- Behavior analysis
- File and registry accesses
- Network activity

#### Soucis de l'heuristic :
- Difficile a setup
- Faux positifs
- Faux négatifs

Donc non.

#### File signature check

Chiffrement et hashage, le chiffrement (encryption) est réversible et contient toute la donnée. Le hashing est non réversible et il contient que peu de la donnée.

L'avantage : c'est rapide, c'est précis et ça se met facilement en place.

On peut donc fouiller une base de donnée connue de fichier verrolés et comporer le hash du fichier.

Le premier obstacle est qu'il faut une base de donnée et la solution peut etre chère.

Tout ceci n'est pas utile car l'application dont parle Alexandre est un BO privé, tout le monde est authentifié et les types de fichiers à upload sont limités.

L'antivirus est trop compliqué mais la sandbox reste réalisable.

Une sandbox est une boite isolé du système dans laquelle on lance un fichier et on voit s'il y a des dégats ou non.

1. Créer un répertoire de sandbox
2. Logger, reset de la sandbox.
Analyse
3. On cherche les caractères interdits dans le nom du fichier. Par exemple, avoir une .php, ".." pour jouer dans l'arborescence ou autres caractères d'échappement.
4. Lecture du fichier, on change les permissions pour retirer les droits d'éxécution.
5. Les octets magiques : en lisant le binaire dump par PHP 8.2 sous macos, on voit quelques octets qui permettent de lire l'éxécutable.

ENDIANNES = boutisme (wtf)

En anglais, la représentation des données binaires dans un processeur.
Cela existe pour une raison historique, d'efficacité et de compatibilité réseau.

On fait donc un "swap endianness" pour la machine ce qui explique l'absence des magic bytes attendus.
    
6. On lis les magic bytes du fichier pour savoir de quel type est le fichier donné en sandbox, en cas d'éxécutable on le jette.

#### Comment gérer les archives

Les archives peuvent contenir une récursion infinie pour faire tomber le serveur. Pour les analyser, on peut utiliser en PHP le \PharData.
On va vérifier la profondeur de l'archive pour vérifier que l'on ne tombe pas dans le cas précédent.

Les dossiers ou fichiers peuvent essayer de traverser des directoires en se décompressants, on doit donc vérifier si dans les noms de fichiers nous avons ".." ou "\" et que ces fichiers essayent d'accéder au système par le disque.

Les vérifications sur les fichiers précédemment évoqués sont aussi appliqués.

Les archives zip ne sont pas capable d'avoir l'information de droit d'éxécutable mais les rar et autres en sont capables, il faut donc s'en prévenir. En application un masque 001 001 001 on peut vérifier si un 
des groupes du fichier ont le droit d'éxécuter un fichier.

(1<<6) | (1<<3) | 1

Dernier check, les tar bombs.

Une archive qui contiendraient des milliers ou plus de fichiers et qui se mettent tous à la source. PharData nous en prémuni donc pas de soucis avec les standards PHP !

### Comment deboguer xdebug - Kevin Dunglas

REX sur la dernière fois ou Kevin s'est retrouvé bloqué devant du code ou une prod sans savoir quoi faire.

Kevin s'est retrouvé à avoir un soucis dans FrankenPHP, sa solution de serveur PHP à haute performance.

Grace à un arbre de décision, il déduit si le bug vient d'une librarie ou du code projet (sans certitude en revanche).
Un autre arbre de décision a été fait, il est du point de vue du mainteneur.

Deux rapports de bug ont été fait sur dunglas/symfony-docker puis laravel/octane. Tous lié à FrankenPHP et Xdebug.

La nouvelle version de PHP mettra Kevin sur une fausse piste (mais si, celle du downgrade!).

Xdebug permet de faire de debug par étape, avoir de meilleurs erreurs, du profiling ou encore du code coverage. Il est intégré à beaucoup d'IDE et dans de nombreuses stack PHP par défaut.

Petit clin d'oeil a Derick Rethans "Better debugging with xdebug" qui permet de mieux comprendre xdebug !

La piste se trouve autour de Xdebug et les builds ZTS, sans stacktrace il est difficile d'avoir des certitudes...

#### Première étape : reproduire le soucis

Malgré la meme version PHP, FrankenPHP et Xdebug, en local le bug ne se produit pas.
L'investigation sera aidé par un simple reproducer d'un utilisateur.

```php
<?php

error_reporting(0);
sleep(1);
phpinfo();

// In the same directory, with xdebug enabled : frankenphp php-server
```

De temps en temps, le bug se produit (oui, de temps à temps..).

#### Deuxième étape : récolter des informations

Si le code se trouvait dans le code PHP, on pourrait utiliser xdebug mais ici on doit débugger le débuggeur... Donc le bug se trouverait du coté du C ou autre.
Le GNU debugger permet de vérifier tout langage compilé, il sera donc utile ici !

Les programmes étant installés en mode production/optimisé, le debugger montrera des caractères illisibles pour l'humain car il manque les "Debug symbols" au lieu d'adresses mémoires..
Une solution simple se trouve de recompiler les sources des logiciels qui posent soucis, ici par exemple PHP avec le moins d'extension possible et si un bug se trouve dans la source, on peut tester son fix directement.

Une demo live du reproducer montrera la stacktrace et une backtrace qui indique dans son cas une récursion infinie, xdebug s'appelle lui meme ce qui produit des soucis de CPU et le crash du programme.

En cherchant dans xdebug, xdebug remplace la fonction error_reporting puis remet la fonction d'origine ensuite. Ce comportement est documenté par PHP pour les extensions, malheureusement xdebug n'avait pas prévu le mode ZTS et donc remplace la fonction remplacée par la fonction remplacée etc... En faisant quelques modifs, le patch a pu se faire facilement et devenir thread safe.
