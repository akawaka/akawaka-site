## Create your own malware sandbox - Alexandre Daubois

Le contexte projet pour la conférence sera celui d'un pentest, le projet d'Alexandre ayant reçu un audit qui demande de mieux sécuriser l'upload de fichier.

3 options :
- Abandonner (pas possible pour le client)
- Une sandbox
- Créer leur propre antivirus

L'option de créer leur propre antivirus est avant tout envisagée, la détection heuristique étant explorée mais ayant trop de défauts pour être implémentée ici.

- Difficile a setup
- Faux positifs
- Faux négatifs

La solution suivante sera donc d'utiliser la signature de fichier.

<img src="/build/front/images/blog/sandbox.jpg">

### File signature check

Une courte introduction au chiffrement et au hashage nous permettra de voir que la solution nécessite soit d'utiliser un service externe connaissant les hash de fichiers verrolés, soit de l'entretenir soit même.

En considérant le fait que l'application est un back office interne, authentifié et privé, une sandbox maison semble être la meilleure solution pour répondre aux exigences du pentest.

Voici les grandes étapes :

- Créer un répertoire de sandbox
- Logger, reset de la sandbox. 
- On cherche les caractères interdits dans le nom du fichier. Par exemple, avoir une .php, ".." pour jouer dans l'arborescence ou autres caractères d'échappement.
- Lecture du fichier, on change les permissions pour retirer les droits d'éxécution.
- Les octets magiques : en lisant le binaire dump par PHP 8.2 sous macos, on voit quelques octets qui permettent de lire l'éxécutable.
- On lit les magic bytes du fichier pour savoir de quel type est le fichier donné en sandbox, en cas d'éxécutable on le jette.

Un petit cas exceptionnel existe, les archives..

### Comment gérer les archives

Toutes les sécurités appliqués aux fichiers sont ici réappliqués à chaque fichier et dossier qu'on retrouve dans l'archive avec l'aide de la classe \PharData de PHP pour vérifier certaines attaques possibles.

- Tar bomb
- Récursion infinie
- Permission de fichiers dans les archives

Il aura fallu moins de 10 classes PHP pour créer cette sécurité.
Voila, la Sandboxed Directory Forbidden characters Magic Bytes with endianness et Archives handling est née. Si vous ne comprenez pas tout ces termes.. retrouvez la rediffusion bientôt sur la chaine de l'AFUP!

## Comment deboguer xdebug - Kevin Dunglas

Cette conférence est un REX sur la dernière fois ou Kevin s'est retrouvé bloqué devant du code ou une prod sans savoir quoi faire.

Kevin s'est retrouvé à avoir un soucis dans FrankenPHP, sa solution de serveur PHP à haute performance.

Grâce à un arbre de décision, il déduit si le bug vient d'une librarie ou du code projet (sans certitude en revanche).

<img src="/build/front/images/blog/kevin-graph-1.jpg">

Un autre arbre de décision a été fait, il est du point de vue du mainteneur.

<img src="/build/front/images/blog/kevin-graph-2.jpg">

Deux rapports de bug ont été fait sur dunglas/symfony-docker puis laravel/octane. Tous liés à FrankenPHP et Xdebug.

La nouvelle version de PHP mettra Kevin sur une fausse piste (mais si, celle du downgrade!).

Xdebug permet de faire de debug par étape, avoir de meilleures erreurs, du profiling ou encore du code coverage. Il est intégré à beaucoup d'IDE et dans de nombreuses stack PHP par défaut.

Petit clin d'oeil a Derick Rethans "Better debugging with xdebug" qui permet de mieux comprendre xdebug !

La piste se trouve autour de Xdebug et les builds ZTS, sans stacktrace il est difficile d'avoir des certitudes...

### Première étape : reproduire le soucis

Le bug n'étant pas reproduit en local, l'investigation était compliquée. Par chance un utilisateur va partager un exemple reproduisible !
```php
<?php

error_reporting(0);
sleep(1);
phpinfo();

// In the same directory, with xdebug enabled : frankenphp php-server
```

Mais le bug se produit de temps en temps (oui, de temps à temps..).

### Deuxième étape : récolter des informations

Si le code se trouvait dans le code PHP, on pourrait utiliser xdebug mais ici on doit débugger le débuggeur... Donc le bug se trouverait du coté du C ou autre.
Le GNU debugger permet de vérifier tout langage compilé, il sera donc utile ici !

Les programmes étant installés en mode production/optimisé, le debugger montrera des caractères illisibles pour l'humain car il manque les "Debug symbols" au lieu d'adresses mémoires..
Une solution simple se trouve de recompiler les sources des logiciels qui posent soucis, ici par exemple PHP avec le moins d'extension possible et si un bug se trouve dans la source, on peut tester son fix directement.

Une demo live du reproducer montrera la stacktrace et une backtrace qui indique dans son cas une récursion infinie, xdebug s'appelle lui même ce qui produit des soucis de CPU et le crash du programme.

En cherchant dans xdebug, xdebug remplace la fonction error_reporting puis remet la fonction d'origine ensuite. Ce comportement est documenté par PHP pour les extensions, malheureusement xdebug n'avait pas prévu le mode ZTS et donc remplace la fonction remplacée par la fonction remplacée etc... En faisant quelques modifs, le patch a pu se faire facilement et devenir thread safe.

Donc :
- Communiquer
- Obtenir des preuves
- Utiliser les bons outils
- Utiliser un moteur de recherche (ou lire des livres!)
- Développeur des théories
- Contribuer !

Ce talk nous aura permis d'avoir les outils pour debugger nos propres programmes efficacement et est un excellent REX étape par étape d'un bug complexe à détecter.

## Les nouveautés de PHP 8.4 - Derick Rethans

### [Property hooks](https://stitcher.io/blog/new-in-php-84#property-hooks-rfc)

On peut définir des hooks (fonction de get/set etc sur les propriétés) afin de définir des getters et setters locaux aux propriétés.

Ils sont utilisés lorsqu'on set/get une propriété public et ont pour but de ne plus avoir de setter/getter dans les classes.

### [new sans parenthèse](https://stitcher.io/blog/new-in-php-84#new-without-parentheses-rfc)

Afin d'éviter de faire des parenthèse lors de l'instanciation d'une classe et l'utilisation d'une de ses fonctions, le new est utilisable sans parenthèse. C'est un sucre syntaxique qui va pouvoir améliorer la lisibilité de notre code.

```php
$name = (new ReflectionClass($objectOrClass))->getShortName();
// vs
$name = new ReflectionClass($objectOrClass)->getShortName();
```

### [Parsing html5](https://stitcher.io/blog/new-in-php-84#new-html5-support-rfc)

Une classe \Dom\HTMLDocument est désormais capable de parser de l'html5 correctement.

### Lazy objects

Les lazy objects vont permettre aux framework et développeurs d'améliorer les performances des objets en créant des proxies qui iront fetch les dépendances uniquement à l'appel de la classe.

[Plus de détail dans cet article de stitcher.io](https://stitcher.io/blog/improved-lazy-loading)

### [JIT amélioré et documenté](https://stitcher.io/blog/new-in-php-84#jit-changes-rfc)

Il est désormais activable via un paramètre donné au lieu d'un type de fonctionnement 
Pour conserver le jit activité sur un système, il va falloir changer cette directive ini.

```ini
opcache.jit=tracing
```

### Subclasses de PDO

Afin d'améliorer les API de PDO, des subclasses par base de donnée existent désormais. Exemple : PDOMySQL, PDOFirebird, PDOCi, PDOODBC etc...

### [Rounding modes (pour compléter ceil, floor)](https://php.watch/versions/8.4/round-new-modes)

La fonction round() disposait de 4 arguments, il en a désormais 8.

```php
round(3.14, precision: 0, mode: PHP_ROUND_HALF_UP); // 3.0
```

Les anciennes méthodes:

PHP_ROUND_HALF_UP: Le classique, 1.5 devient 2 et -1.5 devient -2.
PHP_ROUND_HALF_DOWN: L'inverse, 1.5 devient 1 et -1.5 devient -1.
PHP_ROUND_HALF_EVEN: On arrondit au nombre pair le plus proche, 1.5 et 2.5 deviennent 2.
PHP_ROUND_HALF_ODD: On arrondit au nombre impair le plus proche, 1.5 devient 1 et 2.5 devient 3.

Les nouvelles méthodes en PHP 8.4:

PHP_ROUND_CEILING: On arrondit toujours au nombre entier supérieur, 1.1 et 1.5 deviennent 2, et -1.1 et -1.5 deviennent -1.
PHP_ROUND_FLOOR: On arrondit toujours au nombre entier inférieur, 1.1 et 1.9 deviennent 1, et -1.9 et -1.1 deviennent -2.
PHP_ROUND_TOWARD_ZERO: On arrondit en direction de zéro, 1.9 et 1.1 deviennent 1, et -1.9 et -1.1 deviennent -1.
PHP_ROUND_AWAY_FROM_ZERO: On arrondit en s'éloignant de zéro, 1.1 et 1.9 deviennent 2, et -1.1 et -1.9 deviennent -2.

### Default bcrypt cost factor augmenté pour sécuriser

Puisque les processeurs évoluent, le besoin de sécurité aussi. Afin de mieux protéger les encryptions bcrypt, son facteur par défaut à été augmenté de 10 à 12 itérations.

### [Deprecate implicit null type](https://stitcher.io/blog/new-in-php-84#implicit-nullable-types-deprecation)

Ce code fonctionnait en 8.3 et va désormais lever un deprecated, il est nécessaire de déclarer que $book est nullable.

```php
function save(Book $book = null) {}

// Deprecated: Implicitly marking parameter $book as nullable is deprecated,
// the explicit nullable type must be used instead
``` 

### [Policies & procedure for RFC rfc/policies](https://github.com/php/policies)

La politique des RFC PHP est désormais disponible en ligne et informe sur le processus de release, les RFC pour les nouvelles features, les standards de code, la classification de sécurité et la politique de sécurité et de processus.

### Changement de release cycle 2+2 ans de secutiy support 

Avant, chaque version majeure de PHP (comme PHP 8.3, 8.4, 8.5, etc.) recevait deux ans de corrections de bugs et de sécurité, suivis d'un an de mises à jour de sécurité uniquement.  En gros, chaque version était supportée pendant trois ans, et il fallait mettre à jour son application PHP avant la fin de cette période.

Maintenant, les choses changent !  Chaque version majeure recevra deux ans de corrections de sécurité au lieu d'un.  Cela signifie que chaque version de PHP sera supportée pendant quatre ans au total : deux ans de maintenance active et deux ans de corrections de sécurité.

Pourquoi c'est une bonne nouvelle ?

- Plus de temps pour migrer : Vous avez plus de temps pour mettre à jour vos applications PHP vers une nouvelle version.
- Meilleure stabilité : Vos applications bénéficient d'un support de sécurité prolongé, même si vous n'êtes pas prêt à passer à la dernière version de PHP.

Les features mineurs dans le futur se retrouveront dans les betas. 

### Asymetric visibility

PHP 8.4 introduit une nouvelle fonctionnalité intéressante appelée "visibilité asymétrique" pour les propriétés de classe. En termes simples, cela vous permet de contrôler plus finement l'accès en lecture et en écriture à vos propriétés.

**Avant PHP 8.4:**

Vous pouviez définir une propriété comme `public`, `protected` ou `private`, et ce niveau de visibilité s'appliquait à la fois à la lecture et à l'écriture.

**Avec PHP 8.4:**

Vous pouvez maintenant spécifier une visibilité différente pour la lecture et l'écriture d'une propriété. Par exemple, vous pouvez rendre une propriété `public` en lecture (accessible à tous) mais `private` en écriture (modifiable uniquement à l'intérieur de la classe).

**Comment ça marche ?**

On utilise une nouvelle syntaxe pour définir la visibilité asymétrique :

```php
class MaClasse {
  public private(set) $maPropriete;
}
```

Dans cet exemple, `$maPropriete` est accessible en lecture par tout le monde (`public`), mais seule la classe `MaClasse` peut la modifier (`private(set)`).

**Avantages de la visibilité asymétrique :**

* **Encapsulation améliorée :**  Vous contrôlez précisément comment les propriétés sont utilisées, ce qui rend votre code plus robuste et moins sujet aux erreurs.
* **Flexibilité accrue :**  Vous pouvez créer des propriétés accessibles en lecture seule depuis l'extérieur de la classe, tout en permettant à la classe elle-même de les modifier.
* **Code plus propre :**  Vous pouvez éviter d'écrire des méthodes "getter" et "setter" inutiles pour les propriétés en lecture seule.

**Exemples d'utilisation :**

* **Propriétés immuables :**  Créez des propriétés qui ne peuvent être modifiées qu'une seule fois lors de l'instanciation de l'objet.
* **Variables de configuration :**  Permettez la lecture de variables de configuration depuis l'extérieur de la classe, mais limitez leur modification à la classe elle-même.

La visibilité asymétrique est une fonctionnalité puissante qui vous donne plus de contrôle sur vos classes et améliore la qualité de votre code PHP.


### L'attribut DEPRECATED est amélioré

Effectivement, c'est une précision intéressante qui permet d'indiquer la version de PHP à partir de laquelle l'élément a été déprécié. 

**Exemple :**

```php
#[Deprecated(since: "8.4")]
function ancienneFonction() {
  // ...
}
```

Dans cet exemple, on indique clairement que `ancienneFonction()` est dépréciée depuis PHP 8.4. 

**Avantages de l'argument `since` :**

* **Documentation plus précise :**  On sait exactement quand la dépréciation a été introduite.
* **Meilleur suivi des dépréciations :**  On peut facilement identifier les éléments dépréciés dans une version spécifique de PHP.
* **Migration plus facile :**  Lors de la mise à jour vers une nouvelle version de PHP, on peut repérer plus facilement les éléments à modifier dans le code.

**En résumé :**

L'argument `since` de l'attribut `#[Deprecated]` est un ajout utile qui améliore la gestion des dépréciations en PHP. Il permet de documenter plus précisément le code et de faciliter la migration vers de nouvelles versions.

## Le cycle de vie d'une variable - Benoit VIGUIER

Benoit nous propose un rapide tour d’horizon des pièges et outils à connaître pour optimiser la gestion mémoire de nos programmes.

Composer dans leur commit ont utilisé la fonction "gc_disable()" afin de désactiver le collecteur de reference circulaire du gargabe collector de PHP ce qui a crée une émulsion dans la communauté PHP.

Ils ont pu justifier cet usage exceptionnel est dû au cas de longs scripts en console. Contrairement aux appels web qui sont destinés à peu durer, les appels CLI peuvent durer plus longtemps et mettre à mal le garbage collector de PHP et permettre des fuites mémoires.

Pour optimiser le gargabe collector de PHP nous pouvons employer deux méthodes assez simplement :


### [WeakReference](https://www.php.net/manual/en/class.weakreference.php)

Une classe native pour créer des objets sans incrémenter le nombre de références à une variable.

En prenant cet exempel de code :

```php
class Node {
    public ?Node $parent = null;
    public ?Node $child = null;
}
```

Comme les deux instances de Node se reference elle meme, unset ne va pas libérer la mémoire à cause de la référence circulaire.
Si on crée une WeakReference sur l'un ou l'autre alors la mémoire sera libérée.

### [WeakMap](https://www.php.net/manual/en/class.weakmap.php)

Si on supprime un élément de la map, la map supprime sa référence sur l'élément. Son avantage par rapport au SplObjectStorage est qu'il ne contribue pas aux références mémoires

[Retrouvez ses slides sur son site](https://b-viguier.github.io/downloads/talks/ForumPhp2024_GreatCircleOfLife.pdf).

## PMU: un plugin composer pour la gestion de Monorepository en PHP - Antoine Bluchet

Antoine a présenté une conférence intéressante sur les monorepos en PHP. Il a commencé par expliquer ce qu'est un monorepo : un repository unique contenant plusieurs projets liés. Il a ensuite comparé cette approche aux polyrepos, où chaque projet a son propre repo.

### Avantages du monorepo :

- Visibilité accrue : Tous les projets sont au même endroit, ce qui facilite la compréhension des dépendances et la collaboration.
- Partage de code simplifié : Réutiliser du code entre projets devient trivial.
- Cohérence : Maintien de la même version des dépendances et des outils pour tous les projets.
- Intégration continue/déploiement continu plus efficace : Un seul pipeline pour tous les projets.

### Inconvénients du monorepo :

- Complexité accrue : La gestion d'un monorepo peut être plus complexe, surtout pour les gros projets.
- Performance du VCS : Un monorepo très volumineux peut impacter les performances de votre système de contrôle de version.

### Outils pour gérer les monorepos en PHP :

- Symfony Flex : Permet de gérer les dépendances et les liens symboliques entre les projets.
- splitsh-lite : Outil développé par Fabien Potencier pour extraire l'historique d'un sous-dossier d'un repo Git.
- symplify/monorepo-builder : Package axé sur les releases et Git, avec des commandes pour gérer les dépendances et valider le monorepo.
- Plugin Composer développé par Antoine : Permet de gérer les dépendances, de visualiser le graphe de dépendances, de lancer des commandes sur tous les projets, et de créer des liens symboliques.

### Points importants à retenir :

Le choix entre monorepo et polyrepo dépend du contexte et des besoins du projet.
Il est important de bien définir les relations entre les projets dans un monorepo.
Des outils existent pour faciliter la gestion des monorepos en PHP.

Pour aller plus loin :

- monorepo.tools : [Site web avec des informations sur les monorepos](https://monorepo.tools/).
- Article de blog de Tomas Votruba sur les monorepos : [son article](https://tomasvotruba.com/blog/good-bye-monorepo)
- Plugin Composer développé par Antoine : [son package](https://github.com/soyuka/pmu)

## Éliminer le gaspillage dans votre processus de développement - 

C'était vraiment inspirant de voir comment on peut optimiser notre façon de travailler et livrer plus de valeur, plus rapidement.

Le speaker a bien insisté sur l'importance de la performance collective.  Oubliez le mythe du "programmeur 10x" qui code comme une machine sans se soucier des autres. La clé, c'est la collaboration, le partage des connaissances et l'amélioration continue en équipe.

### Coder ensemble, c'est coder mieux !

D'ailleurs, il a partagé une technique super intéressante : la programmation en mob ou en binôme.  L'idée, c'est de coder à plusieurs, avec un pilote au clavier et des navigateurs qui guident et réfléchissent ensemble. Au début, ça peut paraître bizarre, voire inefficace, mais en réalité, ça permet de réduire les erreurs, de mieux comprendre le code et de prendre des décisions plus éclairées. Et puis, avouez-le, c'est quand même plus sympa de bosser ensemble !

### Fluidifiez votre "chaîne de valeur"

Un autre point important, c'est de livrer de la valeur en continu.  Pour ça, il faut analyser notre "chaîne de valeur" et identifier les points de blocage.  Trop souvent, on se retrouve avec des branches qui stagnent, des features qui n'arrivent jamais en production, des process de review interminables...  Bref, du gaspillage !

Le speaker nous a donné plein de conseils pour fluidifier tout ça : automatiser les tests, supprimer les étapes inutiles, découper les tâches en petits morceaux...  Il a même parlé d'une méthode assez radicale pour le refactoring, la méthode Mikado, qui consiste à avancer par petits pas et à ne rien commiter tant que le code n'est pas parfait.

### Conclusion

En gros, l'idée c'est de simplifier, d'automatiser et de collaborer pour que le code puisse être déployé rapidement et en toute sécurité.
